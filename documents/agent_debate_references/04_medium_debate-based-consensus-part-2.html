Patterns for Democratic Multi‑Agent AI: Debate-Based Consensus — Part 2, Implementation

16 min read

Welcome back!

In Part 1, we laid the groundwork for understanding the Debate-Based Pattern principles and how it can shape the behaviour of multi-agent AI systems. Now, it’s time to roll up our sleeves and dive into the heart of the matter: implementing the Debate-Based Consensus Pattern.

Before we get into the technical details, let’s quickly revisit what this pattern is all about.

In modern AI applications, using multiple AI agents that debate to reach a consensus can produce more robust and accurate results than relying on a single model or a simple voting ensemble. In this architectural pattern, several AI agents engage in a structured discussion — exchanging arguments, critiquing each other, and refining their answers over multiple rounds — until they converge on a final answer (either by true agreement or by a decision rule after a fixed number of rounds). This guide will walk through how to implement this multi-agent debate pattern using two popular frameworks: LangChain and Semantic Kernel.

Understanding the Multi-Agent Debate Pattern

Multi-Agent Debate is an architecture where multiple independent LLM-powered agents collaborate and compete on a task by discussing it amongst themselves. Each agent operates with its own perspective or strategy (often defined by a distinct prompt or role) and can communicate with the others. The process is analogous to a panel of experts deliberating on a question.

  • Decentralized Interaction: There is typically no single omniscient controller micromanaging each step. Instead, agents communicate peer-to-peer or in a round-robin fashion, sharing their answers and critiques with one another. One agent might take a turn to speak, then another responds, and so on.
  • Iterative Rounds: The conversation continues for several rounds. In each round, agents may adjust their answers based on others’ arguments. They might point out inconsistencies in another agent’s answer or bring new evidence to support their own. Over time, you’d expect incorrect or weaker answers to be refined or replaced by stronger arguments.
  • Consensus or Decision: The debate ends when either all agents agree on an answer (unanimous consensus) or a predetermined stopping condition is reached (e.g. a fixed number of rounds). If a hard stop occurs without full agreement, a final answer must be synthesized — for example, the agents could take a final vote, or a designated “judge” agent (or algorithm) can choose the most convincing answer.
  • Comparison to Simple Voting: Unlike the Voting-Based Consensus Pattern (where agents submit answers independently and the majority wins), the debate pattern involves dynamic interaction. Agents are not isolated; they actively influence each other. This can lead to better reasoning as errors get corrected through critique. (However, it also requires more computation and careful orchestration.)

Use Cases: This pattern is useful for complex problems where a single pass might be prone to error. For example, brainstorming and ideation (where multiple creative suggestions are refined into one), verification of reasoning (one agent proposes a solution, another acts as a critic to verify facts and logic), or ethical/policy decisions (agents debate pros and cons from different perspectives). The goal is to leverage multiple viewpoints and the self-correcting nature of debate to improve the quality of the final answer… but also avoid the democracy-paralysis (endless debate and “bureaucracy”…).

Implementing the Debate Pattern with LangChain

LangChain is a framework that enables developers to build applications with LLMs by chaining together prompts, models, and even agents that use tools. It has support for multi-agent workflows, especially via an extension called LangGraph, which allows defining agents as nodes in a conversation graph with cycles (iterative loops). While LangChain’s core provides building blocks (LLMs, memory, agent classes), LangGraph specifically makes it easier to coordinate multiple agents and manage complex control flows (like turn-taking).

Key Components for LangChain Implementation:

LLM Instances: You will need an underlying language model for each agent (for instance, two instances of ChatOpenAI if using OpenAI’s GPT-4 for two agents). Each agent can use the same base LLM or different models if you prefer (even fine-tuned ones for specialized behavior).

Agent Roles (Prompts): Define a distinct system prompt for each agent to shape its persona or viewpoint. For example, one agent could be instructed to be a “risk-taking optimist” and another a “cautious pessimist,” or more simply each agent just tries to solve the problem normally but independently. The prompt serves as initial instructions that remain constant (e.g., a role or ideology), ensuring diversity of thought.

Memory / Conversation State: The agents will communicate over multiple turns, so a mechanism to store the conversation history is needed. LangChain was providing ConversationBufferMemory that now has been deprecated (here the migration guide to the new classes) or you can manage a list of messages manually. Each agent should be able to see the dialogue so far (at least the parts you allow it to see). In a fully shared debate, every agent sees all messages from all agents, emulating everyone hearing each other’s arguments.

Orchestration Loop: You need to implement the turn-taking logic. This can be done manually with a simple loop, or by using utility classes. In LangChain examples, a pattern is to use a Dialogue Simulator that orchestrates agents in a round-robin conversation. For instance, implementing multple DialogueAgent instances (each representing an agent with send/receive methods) and a DialogueSimulator that manages which agent speaks next. The simulator can inject an initial question and then alternate turns between agents automatically.

Consensus Check: After each round (or at the end of each full cycle of all agents speaking), you should check if the debate has converged. This could be as simple as comparing the latest answers of all agents to see if they’re the same (for tasks where a single answer string is sought), or checking a shared flag/keyword in their responses (e.g., they might all say “I agree that …”). Alternatively, decide on a fixed number of rounds and only then determine the outcome.

Step-by-Step (LangChain):

Setup LLMs and Agents: Initialize the language model endpoints you will use (for example, OpenAI API through LangChain’s ChatOpenAI wrapper). Create one LangChain Chain or Agent per participant. A straightforward approach is using ConversationChain for each agent with a system prompt defining its role. Ensure each agent’s chain has access to a memory that will store conversation context (the dialogue).

Define the Debate Routine: If coding from scratch, structure a loop for interaction. For two agents A and B, you could do:

  • Provide the initial question as input to Agent A (or to both if they should both start with an answer).
  • Get Agent A’s answer. Append it to a shared conversation log.
  • Feed the updated conversation (question + A’s answer) into Agent B, so Agent B sees Agent A’s argument, then get Agent B’s response. Append to log.
  • Cycle back to Agent A with the new log (now containing A’s answer and B’s response) so that A can react to B.
  • Continue this alternating process for N rounds or until consensus.

Leverage LangChain Utilities (optional): Instead of manual loops, you can use LangChain’s emerging tools for multi-agent dialogs. For example, LangChain’s LangGraph and the collaboration utilities provide abstractions for this pattern. A DialogueSimulator can automatically handle first steps. You would configure it with your agents and a turn-selection function (round-robin selection) and then simply call something like simulator.inject(initial_message) to start, and loop simulator.step() to run each turn. This yields the speaking agent’s name and message each time, which you can log or analyze.

Consensus or Stopping: Decide when to stop the debate. You might set max_rounds to, say, 5 exchanges each. After each full round, check if the agents’ outputs are effectively identical or if one explicitly concedes. In some implementations, you might include a special token or phrase that agents can output to indicate they want to finalize the answer (e.g., “FINAL ANSWER: …”). If using such a trigger, your loop would watch for it.

Final Answer Synthesis: Once the debate ends, prepare the final answer. If all agents ended up saying the same thing, it’s straightforward — that’s your answer. If there is still disagreement, you have options:

  • Majority Vote: If you have 3 or more agents, you could pick the answer that at least two agents share.
  • Judge Agent: Introduce a separate agent (or one of the existing agents designated as a moderator/judge) to evaluate the arguments and decide. For example, you could feed the transcript to a “judge” LLM and ask which answer seems most convincing or correct.
  • Heuristic Merge: Sometimes you can combine insights from different answers. For example, if each agent contributed part of a solution, you might produce a consolidated answer summarizing the agreed points.

LangChain Example — Two Agents Debating:

To illustrate, let’s consider a simple two-agent debate using pseudo-code with LangChain style. Suppose we want two agents to debate the question: “What are the benefits and risks of AI in healthcare?” — one agent will argue more on benefits, another will focus on risks.

from langchain.chat_models import ChatOpenAI
from langchain.prompts import SystemMessage

# Initialize two chat model instances (e.g., GPT-4 with LangChain wrapper)
llm_a = ChatOpenAI(model_name="gpt-4", temperature=0.7)
llm_b = ChatOpenAI(model_name="gpt-4", temperature=0.7)

# Define roles via system prompts
role_a = SystemMessage(content="You are an AI that strongly highlights the BENEFITS of AI in healthcare.")
role_b = SystemMessage(content="You are an AI that strongly highlights the RISKS of AI in healthcare.")

# Setup conversation history for each agent
conversation = []  # shared conversation log (list of messages)

# Initial question (from a moderator or user)
question = "What are the impacts of AI on healthcare?"
conversation.append({"role": "user", "content": question})

# Have each agent answer initially
response_a = llm_a([ role_a ] + conversation).content
conversation.append({"role": "assistant_A", "content": response_a})

response_b = llm_b([ role_b ] + conversation).content
conversation.append({"role": "assistant_B", "content": response_b})

# Debate rounds
MAX_ROUNDS = 3
for round in range(MAX_ROUNDS):
    # Agent A responds to Agent B
    response_a = llm_a([ role_a ] + conversation).content
    conversation.append({"role": "assistant_A", "content": response_a})
    # Check for consensus or termination keyword
    if "AGREED" in response_a.upper():
        break  # Agent A signaled agreement (just an example condition)
    # Agent B responds to Agent A
    response_b = llm_b([ role_b ] + conversation).content
    conversation.append({"role": "assistant_B", "content": response_b})
    if "AGREED" in response_b.upper():
        break

# After debate, decide final answer:
final_answer = None
if "AGREED" in response_a.upper() or "AGREED" in response_b.upper():
    # If any agent explicitly agreed, use that as consensus (assuming they state it)
    final_answer = conversation[-1]["content"]
else:
    # Otherwise, for simplicity, take the last statement by Agent B as the final answer
    final_answer = conversation[-1]["content"]

print("FINAL ANSWER: ", final_answer)

In this mock-up:

  • Each agent sees the entire conversation (including both agents’ messages) up to that point.
  • They continue to append their messages.
  • We use a simple strategy: if any agent’s message contains the word “agreed” (as an indicator of consensus), we break early. In practice, you’d use a more robust detection or have them output a structured signal like <CONSENSUS_REACHED/>.
  • Finally, we choose the last message as the final answer for demonstration, though you might have more sophisticated logic.

LangChain in Practice: This article, from Cobus Greyling, provides an interesting example with agents debating. In one example, two agents, “AI accelerationist” and “AI alarmist”, debate the impact of automation on employment. They even use tools (like web search) to back up their arguments during the debate. This shows how you can incorporate factual tools into the discussion, making debates not just opinion-based but grounded in retrieved evidence. LangChain’s LangGraph library can formalize such workflows; for instance, you could set up a graph where Agent A and Agent B alternate nodes, passing the conversation state between them on each edge (akin to a state machine ping-ponging between states). The benefit of using LangChain here is flexibility: you can give each agent different abilities (one could have a calculator tool, another a knowledge base tool), and you can instrument the conversation (logging each turn, injecting moderator notes, etc.). Multi-agent designs allow you to divide a complex problem into sub-tasks or perspectives, handled by specialized agents, which can lead to better overall outcomes.

Example Outcome: Running a debate like the one above might produce an insightful final answer. Agent A might list how AI can improve diagnostics and personalize treatment (benefits), Agent B might highlight concerns like data privacy and potential misdiagnosis (risks). Through debate, they could agree that “AI in healthcare offers significant improvements in efficiency and predictive power, but it must be implemented with care to manage risks like bias and privacy.” This agreed statement balances both viewpoints. In scenarios where they don’t naturally converge, you as the developer might take their last statements and combine them (or ask a third agent to summarize the debate). The process provides a richer answer than either agent alone would likely produce initially.

Implementing the Pattern with Semantic Kernel

Semantic Kernel (SK) is another framework (open-sourced by Microsoft) that enables AI integration into applications, with an emphasis on planning, skills (tools), and lately, agents. SK has introduced a first-class Agent Framework to simplify creating and orchestrating multiple agents. As of recent updates, SK supports built-in multi-agent orchestration patterns, making it easier to implement scenarios like our structured debate. One of these patterns is specifically Group Chat Orchestration, which is ideal for an agent debate: it manages a conversation among multiple agents (and even a human optionally) with a coordinator ensuring everyone gets a turn.

Key Components for Semantic Kernel Implementation:

SK Kernel and Agents: First, initialize the Semantic Kernel runtime (the Kernel in Python or KernelBuilder in .NET). Define your agents using SK’s agent abstraction. Each agent will have:

  • A name/ID
  • Instructions (similar to a system prompt defining its persona or role).
  • Optionally, a set of skills/plugins it can use (for instance, an agent might have a Calculator skill or a KnowledgeBase skill depending on what plugins you register, just like tools in LangChain).
  • A model backend (e.g., an OpenAI GPT-4 or Azure OpenAI deployment).

Orchestration Pattern: Choose the GroupChatOrchestration pattern for a debate-style interaction. SK also offers Sequential (pipeline), Concurrent (parallel independent outputs), Handoff (transfer control based on context), and a more complex “Magentic” orchestration for dynamic workflows. For our purposes, Group Chat is the one where agents engage in a conversation together, akin to a chatroom with a moderator.

Orchestrator (Group Chat Manager): In SK’s architecture, a Group Chat manager coordinates the flow of conversation. It decides which agent speaks when. By default, this might be round-robin or can be customized. The manager could also inject a user question to start and even include a human in the loop if needed. Importantly, it keeps track of the shared context (conversation so far) and ensures each agent’s turn sees the latest updates.

Stopping Criteria: SK allows specifying termination conditions. If needed, you can incorporate a rule like stop after X turns or stop when an agent triggers a certain message. (In the current SK API, specifics of customizing the number of rounds may involve using callbacks or a specialized termination skill. If not directly exposed, you may need to programmatically check the content of outputs for an agreement or have an agent serve as the judge to stop the chat.)

Result Aggregation: After the orchestration finishes, SK will provide a result object that likely contains the entire conversation or the final messages from each agent. In a consensus scenario, you might design it such that one agent (or the orchestrator itself) outputs the final agreed answer as the result of the orchestration. Alternatively, you can collate the agents’ last messages similar to the LangChain approach and derive an answer.

Step-by-Step (Semantic Kernel)

Initialize the Kernel and Models: Ensure you have installed the Semantic Kernel packages (and the Agents pack, as needed). Initialize a Kernel instance. Register any plugins if your agents will use additional tools (for example, a web search plugin or calculator plugin). However, a pure debate might not need extra plugins beyond natural language if they’re just talking.

Create Agent Instances: For each debating agent, create an agent object and provide each agent with a descriptive Instructions prompt, similar to LangChain’s system prompt. In Python SK, this might look like:

kernel = Kernel()
agentA = await kernel.create_chat_agent(name="AgentA", instructions="Your persona/instructions here", model=... )
agentB = await kernel.create_chat_agent(name="AgentB", instructions="Your persona/instructions here", model=... )

(Pseudo-code for illustration; the actual API might use classes like ChatCompletionAgent or create_chat_agent with appropriate parameters)

Configure Group Chat Orchestration: SK provides classes for orchestration. For example:

from semantic_kernel.orchestration import GroupChatOrchestration, InProcessRuntime
orchestration = GroupChatOrchestration(members=[agentA, agentB])
runtime = InProcessRuntime()
runtime.start()

This sets up a group chat with the two agents. You could include more in the list. The InProcessRuntime runs the orchestration in the current process (alternatively, SK can orchestrate in other ways, but this is simplest for local usage).

Invoke the Orchestration with a Task: The “task” here is essentially the initial user query or instruction that kicks off the debate. For instance:

result = await orchestration.invoke(task="What are the impacts of AI on healthcare?", runtime=runtime)
final_output = await result.get()

This call will cause the GroupChat manager to send the task to the agents and handle the ensuing conversation, iterating turns between AgentA and AgentB automatically. Under the hood, AgentA might produce a response, then AgentB sees that and responds, and so forth, similar to our manual loop earlier. The orchestrator may run until it determines the conversation is complete. The final_output would contain the final answer (for example, it might be the last message in the chat or a concatenation of responses depending on implementation). In .NET, the pattern is analogous: you instantiate GroupChatOrchestration with agents and call InvokeAsync.

Access and Use the Result: Print or return the final_output. It might be a string with the agreed answer or a full transcript. If it’s a transcript and you need just one answer, you might parse it or design your agents such that one of them (say AgentA) is tasked with outputting the final consensus at the end.

Because SK’s GroupChat orchestration inherently handles multi-turn exchanges, you might not explicitly write a for-loop as we did in LangChain; the framework does it for you. The coordination (whose turn it is, sharing messages, etc.) is built-in, making it very convenient. The developer mostly focuses on defining the agents and their roles, and then letting them converse.

For example, suppose AgentA is given instructions to champion the pros of a subject and AgentB the cons, similar to before. With the SK orchestration, both agents would be loaded with those instructions. The group chat orchestrator could be started with the initial question. The output we get could be something like a combined log or the last agent’s message. If we want a specific format (like a single consolidated answer), we might incorporate that into one agent’s behavior (e.g., one agent could be slightly in charge of concluding the discussion).

Semantic Kernel Example — Pseudo-code (Python-like for clarity):

from semantic_kernel import Kernel
from semantic_kernel.agents import ChatCompletionAgent
from semantic_kernel.orchestration import GroupChatOrchestration, InProcessRuntime

# Initialize kernel and agents
kernel = Kernel()
# Create two agents with OpenAI GPT-4 (assuming appropriate config already set up)
agentA = ChatCompletionAgent(kernel=kernel, name="Advocate", instructions="You strongly advocate the positive impacts of AI in healthcare.")
agentB = ChatCompletionAgent(kernel=kernel, name="Skeptic", instructions="You focus on the potential risks and downsides of AI in healthcare.")

# Setup group chat orchestration
orchestration = GroupChatOrchestration([agentA, agentB])
runtime = InProcessRuntime()
await runtime.start_async()

# Start the debate with a user question
task = "Debate the question: 'Is AI beneficial or harmful to healthcare, and why?'"
result = await orchestration.invoke(task=task, runtime=runtime)
output = await result.get_value_async()

print("Debate Result:", output)
await runtime.stop_when_idle()

(Note: The above is illustrative; the actual SK API method names might differ slightly, such as await agentA.create_thread() or similar, following SK’s official samples. Refer to SK’s documentation for exact syntax. The concept remains: define agents, pick orchestration, run it.)

With SK, you benefit from a unified interface for different patterns. If you wanted to compare a debate (GroupChat) vs a simple parallel answer (Concurrent orchestration) vs a pipeline (Sequential), you can switch the orchestration with one line change. For example, change GroupChatOrchestration to ConcurrentOrchestration to have agents answer independently and aggregate results, which is more like voting. SK’s flexible design thus makes it easy to experiment with this debate vs other decision protocols within the same framework.

Agent Communication in SK: Under the hood, when using GroupChatOrchestration, the mechanism ensures that each agent sees a shared conversation history and has a fair chance to contribute. It’s essentially implementing the turn-taking loop for you. The developer doesn’t need to manually propagate each message to all agents; the orchestrator does that. As the SK blog explains, group chat orchestration is “powerful for simulating meetings, debates, or collaborative problem-solving sessions”. You can even include a human as one of the participants if you want a mixed human-AI debate, and the orchestrator can prompt the human at the right times.

Finalizing the Answer: You might design the debate such that one of the agents is tasked with writing the final consensus. For instance, you could introduce a third agent that acts as a moderator/secretary whose job is to listen to the debate and at the end produce a summary or final recommendation. SK’s orchestration would allow this third agent to chime in last with a summary. Alternatively, you stick to two agents and rely on them naturally converging. If they do not converge, you can examine the output or entire transcript and then apply a rule (e.g., if output contains both viewpoints, you might append a concluding sentence outside of SK).

Example Outcome with SK: The result might look like: “After discussion, we conclude that AI in healthcare is overall beneficial, but we must address data security and bias issues to ensure patient safety.” This single sentence could be produced by one of the agents summarizing both sides. The advantage of SK is that you didn’t have to write the loop logic yourself; you declared what you wanted (a group debate) and it orchestrated the conversation via the LLMs.

LangChain vs Semantic Kernel: A Quick Comparison

Both frameworks can achieve the multi-agent debate pattern, but they offer different levels of abstraction and convenience:

In practice, you might choose LangChain if you need fine-grained control or are already using it for things like agent tool use in other parts of your app. Semantic Kernel is a good choice if you want quick setup with robust patterns, especially in a .NET environment or if you like the plugin-based approach and plan to mix in other enterprise.
